Introduction
3
•Integrated Security Approach: The system combines financial fraud detection and network 
security into a single AI-driven solution, addressing the problem that modern cyber threats 
often span multiple stages such as phishing, network intrusion, and financial theft, which are 
usually handled by separate tools.
•Multi-Domain Behavioral Analysis: It analyzes behavioral patterns across financial 
transactions, network traffic logs, and URL structures to identify suspicious and malicious 
activities that may not be visible when these data sources are examined independently.
•Machine Learning-Based Detection: A Random Forest classifier is used to learn complex 
patterns from the combined datasets, enabling accurate and reliable classification of malicious 
activities while remaining efficient and interpretable.
•Real-Time Defense Console: The model is deployed within a real-time Defense Console that 
provides continuous monitoring, alerts, and actionable insights to help security teams detect, 
analyze, and respond to threats quickly and effectively.

Project Title: A Hybrid Deep Learning Ensemble Model for Credit Card Fraud Detection (2024) 
Objective: To capture both spatial and time-series patterns in transaction logs for improved fraud 
detection. 
Methodology:
•Stacked ensemble model combining CNN (spatial) and LSTM (temporal). 
•XGBoost used as a meta-learner to make the final prediction. 
Key Findings:
•Achieved 97.2% AUC, outperforming standalone models. 
•XGBoost effectively corrected errors made by the deep learning layers. 
Limitations:
•Complex architecture requires significant training time. 
•"Black-box" nature reduces explainability. 
Relevance to My Project: Validates the use of XGBoost as a robust core engine for our Financial 
Bot detection module.

Research Gaps
13
•Security Architectures: Existing frameworks typically address financial fraud, network intrusion, 
and phishing in isolation, failing to detect multi-stage attack chains that span across these domains.
•Lack of Strategic Attribution: Current systems detect "anomalies" but fail to attribute the intent, 
unable to distinguish between low-level automated bots and sophisticated, high-persistence State-
Sponsored actors.
•Latency vs. Accuracy Trade-off: Deep learning models (like CNN/LSTM) offer high accuracy 
but incur high computational costs, making them unsuitable for real-time defense consoles that 
require instant blocking capabilities.
•Missed Behavioral Signatures: Traditional financial models focus on mathematical outliers 
rather than behavioral patterns, often missing specific automated threats like "Zero -Balance 
Draining" where bots empty accounts with robotic precision.
•Explainability Issues: Many advanced detection systems operate as "black boxes," lacking the 
interpretability needed for security analysts to trust and validate alerts in a high -stakes 
environment.
===NEW PAGE===
Problem statement
14
Traditional cyber defense systems operate in isolated silos, treating financial fraud, network 
intrusions, and phishing threats as separate events, which creates a critical blind spot for 
detecting complex, multi-stage attack chains where a phishing email leads to a network breach 
and subsequent financial theft. Furthermore, existing detection models fail to distinguish the 
strategic intent of the adversary, often treating automated 'Zero-Balance' bots and sophisticated 
state-sponsored actors identically, while current banking rules miss behavioral signatures such as 
bots calculating exact balances to drain accounts. Additionally, the reliance on offline analysis or 
complex deep learning models prevents real-time deployment, resulting in reactive measures that 
address breaches only after significant damage has occurred. Therefore, there is a critical need 
for an interpretable, cross-domain "Defense Console" that correlates financial, network, and URL 
patterns to detect, attribute, and mitigate threats in real-time.

•To Detect Behavioral Anomalies: Develop hybrid machine learning models (XGBoost) specifically

•To Enhance Model Explainability: Utilize interpretable tree-based models (Random Forest/Decision 
Trees) to provide clear reasoning for alerts, addressing the "Black Box" limitation of existing deep 
learning systems.

Methodology cont…
19
Data Collection:
The process begins by aggregating data from three distinct domains to build a 
unified defense system.
• Financial Logs: We utilize the PaySim Synthetic Financial Dataset (~6 million 
records) to simulate mobile money transactions. This dataset provides critical 
attributes like type, amount, oldBalanceOrg, and isFraud.
• Network Traffic: We collect Global Cyber-Attack logs (~40,000 incidents) 
containing Source IP, Packet Length, and Attack Type (e.g., DDoS, APT) to 
train the network intrusion module.
• URL Data: We gather a Phishing Website URLs Dataset (~500,000 entries) 
labeled as phishing or benign to train the entry-point scanner.
===NEW PAGE===
Methodology cont…
20
Data Preprocessing:
Before training, the raw data undergoes cleaning to ensure quality and consistency.
• Cleaning: We remove duplicate entries and handle null values, particularly in the 
network logs, to prevent model bias.
• Formatting: Timestamps are standardized to allow for cross-referencing between 
network events and financial transactions.
• Encoding: Categorical variables (like Transaction Type: "CASH-OUT" or 
Protocol: "TCP/UDP") are converted into numerical formats using One-Hot 
Encoding or Label Encoding so the machine learning models can process them.
===NEW PAGE===
Methodology cont…
21
Model Training:
We employ a hybrid "Ensemble" approach, training different models for specific tasks:
• Weighted XGBoost: Trained on financial logs to detect "Zero-Balance" anomalies. We use 
class weighting to prioritize fraud detection, addressing the imbalance where fraud cases are 
rare compared to normal transactions.
• Graph Neural Network (GNN): We treat transactions as a graph to detect "Fraud Rings," 
capturing the flow of money between multiple related accounts rather than viewing them in 
isolation.
• Support Vector Machine (SVM): Trained on URL features to create a high-precision boundary 
between "Bot" and "Human" traffic.
• K-Means Clustering: Used for unsupervised learning to group normal transactions; anything 
falling outside these clusters is flagged as a "New/Unknown" threat.
===NEW PAGE===
Methodology cont…
22
Model Evaluation:
We strictly evaluate the models using metrics relevant to security:
Recall: Our primary metric. We prioritize high recall to ensure we catch every 
instance of account draining.
Latency: We measure the inference time to ensure the system processes transactions 
in < 200 milliseconds, meeting the "Real-Time" requirement.
False Positive Rate: We cross-reference alerts (e.g., checking if a financial anomaly 
is linked to a high-risk URL) to minimize false alarms.
===NEW PAGE===
Methodology cont…
23
Hyperparameter Tuning:
If the model performance (specifically Recall) is below our threshold:
• We perform tuning on the XGBoost and Random Forest parameters.
• This involves adjusting the "learning rate," "tree depth," and specifically the 
"scale_pos_weight" to penalize missing a fraud case more heavily than misclassifying a 
normal one.
Deployment (Defense Console):
Once the "Best Model" is selected:
• The models are deployed into a FastAPI backend.
• They feed into a Real-Time Defense Console (built with React/Streamlit) that visualizes 
threats and includes an automated "Kill Switch" to freeze funds and block IPs instantly

•Source: Mendeley Data / Kaggle 
•Size: ~500,000 URLs (Labeled Phishing vs. Benign).
•Relevance: Trains the Entry-Point Scanner to detect and block malicious links before they 
execute.

Modules description
• System Architecture: The system utilizes a multi-tier architecture, 
transitioning from a React-based Frontend Tier (User Dashboard) to a 
Python-based Backend Tier (FastAPI).
• Cross-Domain Integration: The modules are structurally linked through a 
central FastAPI Application that manages requests between the interfaces 
(Phishing Check, Bot Detector, Fraud Ring Visualizer) and the underlying 
ML Models Engine.
• Unified Flow: The connectivity ensures that URL strings and JSON 
requests (network/financial logs) are processed by specialized models 
before being correlated for a final security verdict.
• Data Persistence: A MongoDB / PaySim DB instance is connected to the 
backend to store real-time logs and facilitate historical behavioral analysis.
===NEW PAGE===
Modules description cont…
Frontend Tier (User Interface)
This layer acts as the entry point for security analysts to interact with the 
system.It is built using React and Streamlit to ensure a responsive, real-time 
experience.
• User Dashboard: A centralized interface that visualizes real-time traffic, 
active "Fraud Rings," and blocked IPs.
• Phishing Check Interface: Allows the user to input specific URL strings 
to be scanned for malicious patterns.
• Bot Detector Interface: Enables the investigation of specific Transaction 
IDs to identify automated behavioral anomalies.
• Fraud Ring Visualizer: Utilizes graph queries to display interconnected 
fraudulent accounts identified by the intelligence engine.
===NEW PAGE===
Modules description cont…
Backend Tier (Orchestration & Storage)
This layer serves as the "brain" of the system, managing the routing of data and logic 
between the user and the machine learning models.
• FastAPI Application: A high-performance Python framework used for request 
routing, handling JSON requests, and orchestrating the security pipeline.
• Data Preprocessing Sub-layer: * Cleaning & Encoding: Uses NumPy and Pandas 
to handle null values and convert categorical variables (like transaction types) into 
numerical formats via One-Hot Encoding.
• Feature Engineering: Transforms raw data into "Cleaned Features" (e.g., URL 
NLP features or balance errors) required for model inference.
• Storage Layer: * MongoDB / PaySim DB: A document-oriented database used to 
store incident logs, flagged IPs, and the PaySim synthetic financial records

Intelligence Engine (Machine Learning)
This layer performs the core predictive analysis using a hybrid ensemble approach.
Phishing Engine:
Algorithms: Employs Support Vector Machines (SVM) and NLP (Natural Language Processing).
Role: Analyzes URL structures (length, keywords) to detect malicious intent before network intrusion 
occurs.
Financial Bot Engine: 
Algorithms: Uses Weighted XGBoost and Graph Neural Networks (GNN).
Role: Predicts the presence of automated bots using an ML model to identify robotic behavioral 
signatures. It specifically identifies "Fraud Rings" where money moves through multiple related 
accounts.
Network Engine: 
Algorithms: Utilizes Random Forest and K-Means Clustering.
Role: Performs traffic analysis to attribute threat actors, distinguishing between low-level bots and 
sophisticated State-Sponsored entities

Data flow cont…
• Preprocessing Pipeline: Data flows from collection into a cleaning stage where 
One-Hot Encoding (via NumPy and Pandas) is applied to categorical variables like

Control flow diagram
• Sequential Logic: The system initiates with URL and Network validation; if these 
entry-point scanners flag an anomaly, the control shifts to the Automated Bot 
Search.
• Dynamic Decision Path: Using Scikit-learn and TensorFlow/PyTorch frameworks, 
the system evaluates model performance; if Recall falls below the required 
threshold, the flow loops back for Hyperparameter Tuning via 
RandomizedSearchCV.

malicious IPs.

SHAP or LIME (XAI tools) to ensure the logic behind the ML prediction is 
interpretable for security analysts.
